# Синхронизация результатов osu! multiplayer

## Цель

Дать хосту турнира возможность привязать osu! multiplayer lobby к матчу турнира по `mp link`, автоматически считать счет назначенных игроков только по картам маппула стадии и надежно обновлять результат до закрытия lobby или ручной остановки синхронизации.

## Принятые ограничения

- У стадии может быть только один маппул.
- В матче участвуют не более двух назначенных игроков, как уже ограничено текущим DTO.
- Игроки сопоставляются с результатами osu! по `users.osu_id`, а не по имени.
- WebSocket для отображения результата не используется.
- Ручной счет разрешен только после закрытия lobby или ручной остановки синхронизации.
- После завершения синхронизации итог заморожен и не пересчитывается при последующих изменениях маппула.

## Выбранный подход

Polling выполняется отдельным NestJS-модулем внутри backend, а PostgreSQL хранит durable-состояние заданий и координирует реплики. Это использует уже имеющуюся инфраструктуру, не требует включения NATS и позволяет позднее перенести модуль в отдельный worker.

Отдельный worker сейчас не создается: новый deployment-контур не нужен для выполнения требований. NATS JetStream также не используется, поскольку периодический polling и восстановление расписания проще и надежнее выразить через состояние в PostgreSQL.

## Компоненты

### Osu match client

Адаптер над установленным `osu-api-extended` принимает числовой osu match ID и возвращает нормализованный снимок:

- время закрытия lobby;
- полный список игр с учетом пагинации API;
- ID игры и карты;
- время окончания игры;
- результаты игроков: osu user ID и score.

Остальной модуль не зависит от типов сторонней библиотеки. Детали авторизации и API остаются внутри существующего инфраструктурного osu-модуля.

### Result calculator

Чистая функция получает снимок osu match, два osu user ID и множество допустимых osu beatmap ID. Она возвращает очки обоих игроков.

Для каждой игры:

1. Игра без времени окончания игнорируется.
2. Карта, отсутствующая в маппуле стадии матча, игнорируется.
3. Результаты посторонних игроков игнорируются.
4. Игра без результатов обоих назначенных игроков игнорируется.
5. При равных score очко не начисляется.
6. Игрок с большим score получает одно очко.

Расчет каждый раз выполняется по полной истории, поэтому повторная обработка одного снимка идемпотентна и не требует отдельной таблицы игр.

### Sync repository

Репозиторий инкапсулирует SQL получения заданий, lease и транзакционную запись результата. Он является единственной частью sync-модуля, знающей текущую схему Drizzle.

### Sync runner

Runner использует нативный таймер Node.js, атомарно забирает ограниченную пачку заданий и обрабатывает их. NestJS scheduler и новая очередь не добавляются.

Runner не вызывает контроллеры или `MatchService`. Его зависимости ограничены osu match client, result calculator и sync repository, поэтому модуль можно перенести в worker без изменения HTTP API.

## Модель данных

Добавляется таблица `match_osu_sync`:

- `match_id` — primary key и foreign key на `matches.id` с `ON DELETE CASCADE`;
- `osu_match_id` — числовой ID lobby с unique index;
- `status` — `active`, `stopped` или `completed`;
- `next_sync_at` — время следующей попытки;
- `lease_until` — срок владения заданием;
- `lease_token` — уникальный токен текущего владельца;
- `last_synced_at` — время последней успешной записи результата;
- `last_error` — последняя ошибка для диагностики хостом;
- `attempts` — число последовательных неуспешных попыток;
- стандартные `created_at` и `updated_at`.

На `mappools.stage_id` добавляется unique index, закрепляющий инвариант «одна стадия — один маппул» на уровне БД.

Существующие `match_participants.score` и `match_participants.is_winner` остаются источником результата для расписания. Новая копия итогового счета не создается.

## Жизненный цикл

### Запуск

При первом назначении корректного `mpUrl` или его фактическом изменении backend извлекает ID из официального `https://osu.ppy.sh/community/matches/{id}` или `https://osu.ppy.sh/mp/{id}` и в той же транзакции активирует sync-задание. Query, fragment, другой host и дополнительные сегменты пути не принимаются. Простое сохранение других полей с неизменным URL не перезапускает остановленный sync.

Явный start endpoint позволяет повторно запустить тот же lobby после остановки. Изменение URL сбрасывает `last_error`, `attempts`, lease и переводит задание в `active`.

### Polling

Каждая backend-реплика периодически выполняет одну атомарную операцию claim: выбирает наступившие `active` задания через `FOR UPDATE SKIP LOCKED`, назначает им новые `lease_token/lease_until` и получает строки для обработки.

После ответа osu API runner начинает транзакцию, блокирует sync-строку и проверяет, что статус по-прежнему `active`, а `lease_token` принадлежит ему. Затем он заново считает результат и обновляет обоих участников. Это fencing-проверка: запоздавший ответ не может перезаписать ручной счет после stop или результат другого владельца lease.

Если osu match содержит `end_time`, результат записывается вместе со статусом `completed`; такое задание больше не выбирается. Иначе очищается lease и назначается следующий polling.

### Остановка и ручной счет

Stop endpoint транзакционно переводит sync в `stopped` и очищает lease. Ручное изменение счета разрешено только при `stopped`, `completed` или отсутствии sync-задания. При `active` backend отвечает `409 Conflict`.

Ручные очки должны быть целыми неотрицательными значениями. `is_winner` выводится из очков: единственный больший счет получает `true`, проигравший — `false`; при ничьей или неполном счете оба значения остаются `null`.

Удаление `mpUrl` останавливает активный sync. Оно не удаляет уже записанный счет.

## Ошибки и восстановление

- Некорректный формат URL отклоняется на HTTP-границе и не создает задание.
- Ошибка osu API записывается в `last_error`; `attempts` увеличивается, а `next_sync_at` получает ограниченный exponential backoff.
- Успешная попытка очищает ошибку и счетчик попыток.
- После падения процесса незавершенное задание снова доступно после `lease_until`.
- Несколько реплик могут работать одновременно: `SKIP LOCKED` исключает двойной claim, а lease token защищает от поздней записи.
- Неизвестный или недоступный lobby продолжает повторяться с максимальной задержкой, пока хост не исправит URL или не остановит sync; временный сбой не переводит задание в необратимое состояние.

Интервал polling, срок lease, размер пачки и максимальная задержка backoff задаются environment variables с безопасными значениями по умолчанию. Это эксплуатационные параметры, а не отдельная бизнес-конфигурация.

## HTTP API и авторизация

- Существующие create/update endpoints матча управляют запуском при назначении или изменении `mpUrl`.
- `POST /tournaments/:id/matches/:matchId/sync/start` повторно активирует сохраненный lobby.
- `POST /tournaments/:id/matches/:matchId/sync/stop` останавливает его.
- Публичный schedule DTO возвращает `syncStatus` и `lastSyncedAt`.
- Отдельный host-only `GET /tournaments/:id/matches/:matchId/sync` возвращает состояние, `lastSyncedAt` и `lastError` для формы редактирования.

Все операции используют существующую tournament policy: изменять ссылку, start/stop и ручной счет может только пользователь с правом управления турниром.

## Отображение на frontend

Публичное расписание продолжает читать `score/isWinner` из REST API. Пока среди отображаемых матчей есть `syncStatus === 'active'`, страница периодически инвалидирует schedule-запрос. Timer очищается при отсутствии активных синков и при уничтожении компонента.

В форме редактирования хост видит статус, время последнего успешного обновления и последнюю ошибку, а также доступные по состоянию кнопки start/stop. Ручные поля счета блокируются при `active` и становятся доступны при `stopped/completed`.

## Проверки

### Unit

- победы обоих игроков и суммирование нескольких игр;
- ничья;
- карта вне маппула;
- незавершенная игра;
- отсутствие результата одного участника;
- посторонний игрок;
- повторный полный расчет того же снимка.

### Persistence и concurrency

- две реплики не получают одну действующую lease;
- истекшая lease доступна другой реплике;
- неверный lease token не может записать счет;
- stop во время API-запроса предотвращает позднюю запись;
- запись результата и переход в `completed` атомарны;
- ошибка планирует повтор с backoff.

### HTTP и frontend

- проверка формата mp URL и прав хоста;
- start/stop переходы;
- `409` для ручного счета во время active sync;
- корректный winner для ручного счета;
- frontend polling запускается только при `active` и всегда очищает timer.

## Критерии готовности

- Счет содержит только победы на завершенных играх с картами единственного маппула стадии.
- Ничьи, посторонние карты, игроки и неполные игры не влияют на счет.
- Счет обновляется до закрытия lobby или stop и после этого автоматически не меняется.
- Хост может вручную изменить итог только при неактивном sync.
- После падения экземпляра синхронизация восстанавливается автоматически.
- При нескольких репликах одна версия задания не обрабатывается конкурентно.
- Frontend показывает актуальные REST-данные без WebSocket.
